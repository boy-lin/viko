# PTS 与 current_pos 差异分析

## 问题描述

从日志中可以看到：
- **最后一个数据包的 PTS**: 123.02s
- **当前播放位置 (current_pos)**: 111.28s
- **差异**: 约 12 秒

## 原因分析

### 1. PTS (Presentation Time Stamp)

- **PTS** 是数据包的时间戳，表示这个数据包应该在什么时候播放
- 当数据包迭代器处理到最后一个数据包时，PTS 是 123.02s
- 这意味着**解码位置**已经到达文件末尾

### 2. current_pos (当前播放位置)

- **current_pos** 是基于 `audio_clock` 计算的当前播放位置
- `audio_clock` 的计算公式：`start_audio_pts + played_samples / sample_rate`
- `played_samples` 是**实际播放**的样本数（从缓冲区取出的样本）
- 所以 `audio_clock` 反映的是**实际播放位置**，而不是解码位置

### 3. 为什么会有差异？

这是**正常的缓冲区积压现象**：

1. **解码速度快**：
   - 数据包被快速解码并放入缓冲区
   - 解码线程可以快速处理所有数据包
   - 当最后一个数据包（PTS=123.02s）被解码时，解码位置已经到达文件末尾

2. **播放速度慢**：
   - 播放受采样率限制（如 44100 Hz）
   - 播放速度是固定的，不能超过实时速度
   - 当解码位置到达 123.02s 时，实际播放位置只有 111.28s

3. **缓冲区积压**：
   - 缓冲区中有 95,744 样本（单通道）
   - 如果采样率是 44100 Hz，这相当于 95,744 / 44100 ≈ **2.17 秒**的音频
   - 但实际差异是 12 秒，说明缓冲区中还有更多数据

### 4. 计算验证

从日志中：
- **已处理样本**: 5,415,936
- **缓冲区样本**: 95,744（单通道）
- **音频时钟**: 111.28s

如果采样率是 44100 Hz：
- 已处理样本 5,415,936 ≈ 122.8 秒（解码位置）
- 缓冲区 95,744 样本 ≈ 2.17 秒
- 实际播放样本 = 5,415,936 - 95,744 = 5,320,192 ≈ 120.6 秒

但音频时钟只有 111.28s，说明：
- 实际播放的样本数 = 111.28 * 44100 ≈ 4,907,448
- 缓冲区中还有 = 5,415,936 - 4,907,448 = 508,488 样本 ≈ **11.5 秒**

## 关键问题

**播放完成判断应该基于什么？**

当前代码在数据包迭代器结束时，使用 `audio_clock`（实际播放位置）来判断是否完成，这是**正确的**。

但问题是：
- 当数据包迭代器结束时，`audio_clock` 只有 111.28s
- 但总时长是 123.04s，还差约 12 秒
- 缓冲区中还有约 12 秒的数据未播放

所以，**播放完成判断应该等待缓冲区播放完毕**，而不是在数据包迭代器结束时立即判断。

## 解决方案

播放完成判断应该：
1. **等待缓冲区播放完毕**（`buffer_samples == 0`）
2. **检查音频时钟是否接近总时长**（`audio_clock >= audio_duration * 0.99`）
3. **确保所有数据包都已处理**（`samples_processed >= expected_samples`）

当前的判断逻辑已经考虑了这些因素，但可能需要调整阈值或等待逻辑。
